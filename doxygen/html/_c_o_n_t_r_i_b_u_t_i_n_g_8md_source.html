1 # Contributing
2 
3 Code must be put into the repository as the result of a pull request, we don't want (unless in exceptional circumstances)
4 people randomly throwing code into the repo.
5 
6 ## Getting Started
7 
8 Make a fork of this repository and clone it to your computer.
9 
10 Make a new feature branch based off of the develop branch (git-flow is your friend).
11 
12 Feature branches should have a lowercase name and be of the form:
13 
14 ```
15 feature/awesome_stuff
16 ```
17 
18 Hack away until your heart is content.
19 
20 Take a look at the [issues](https://github.com/HarveyHunt/seprcph/issues) for ideas on what to work on.
21 
22 ## Code Style
23 
24 Follow [PEP8](http://legacy.python.org/dev/peps/pep-0008/) as closely as sanely possible.
25 
26 Running a code linter, such as [PyFlakes](https://launchpad.net/pyflakes) is always a nice idea- try to minimise errors.
27 
28 The hard rules are as follows:
29 
30  * 4 spaces per tab
31  * use_of_underscores_for_functions_variables_etc
32  * Code should be pythonic
33  * All new code *must* be documented
34 
35 ## Commit Rules
36 
37 Try to stick to the following rules:
38  * Keep the first line of a commit message less than 80 characters.
39  * Keep commit messages descriptive but short.
40  * If you fix an issue or your commit is related to an issue, read more [here](https://github.com/blog/957-introducing-issue-mentions).
41  * Commit small and often, otherwise tools such as [git-bisect](http://git-scm.com/docs/git-bisect) will become useless.
42 
43 ## What comes after code?
44 
45 You have written an awesome new feature and want to get it merged straight away- have you forgotten something?
46 
47 Documentation is an important part of any project, Project Homeslice is no different. Document all new features, as has been done
48 in the README already.
49 
50 If you're feeling especially kind and in a Python mood, try to add some unit tests for your new feature- or even more parts of Project Homeslice in general.
51 
52 ## Sending PRs
53 
54 Once you have made some cool changes, push to github and send a PR.
55 
56 Make sure that the PR is targeted at the develop branch, we want to keep master "clean" and only push to it at the end of a sprint.
57 
58 **Include a link to the Jenkins build associated with your PR.**
59 
60 ## Merging PRs
61 
62 We all have the ability to merge PRs, that doesn't mean we should randomly start merging stuff.
63 
64 At least one person (other than the PR's author) should "sign off" on the PR by commenting on its issue.
